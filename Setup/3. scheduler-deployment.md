# AI Scheduler Deployment Guide
> **Role:** DQL-based intelligent router — selects the best edge node every 30s and forwards inference requests to it. Falls back to Cloud VM if all edge nodes are overloaded.

### Key Parameters

| Parameter | Value | Description |
|-----------|-------|-------------|
| Scheduler Image | `amogh0709/ai-scheduler:v2` | DQN-based router |
| Container Port | `5001` | Scheduler listens here |
| NodePort | `30084` | Exposed on the K8s node |
| Check Interval | `5s` | How often scheduler polls edge `/metrics` |
| CPU Threshold | `0.80` | Edge node CPU limit (80%) |
| MEM Threshold | `0.85` | Edge node memory limit (85%) |
| QUEUE LIMIT | `0.60` | Queue Length limit (60%) |
| Cloud Fallback | `FLOATING_IP:8080` | Used when all edges are overloaded |

## Part 1 — Create Deployment File

The deployment uses direct `env` variables (not a ConfigMap) so all routing config is visible in one place.

### Environment Variables Explained

| Variable | Value | Purpose |
|----------|-------|---------|
| `BACKEND_URLS` | Comma-separated NodePort URLs | All 9 edge nodes the scheduler can route to |
| `CLOUD_SERVER_URL` | `http://FLOATING_IP:8080` | Cloud VM backup — used when all edges are busy |
| `CHECK_INTERVAL` | `5` | Seconds between edge `/metrics` polls |

> **BACKEND_URLS** maps directly to the edge NodePorts from `edge-deployment.yaml`. Port `30084` is reserved for the scheduler itself — that's why edge node 4 uses `30085`.

```bash
gedit scheduler-deployment.yaml
```

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: scheduler
spec:
  replicas: 1
  selector:
    matchLabels:
      app: scheduler
  template:
    metadata:
      labels:
        app: scheduler
    spec:
      containers:
      - name: scheduler
        image: amogh0709/ai-scheduler:v2
        imagePullPolicy: Always
        ports:
        - containerPort: 5001
        env:
        - name: BACKEND_URLS
          value: "http://172.16.23.129:30081,http://172.16.23.129:30082,http://172.16.23.129:30083,http://172.16.23.129:30085,http://172.16.23.129:30086,http://172.16.23.129:30087,http://172.16.23.129:30088,http://172.16.23.129:30089,http://172.16.23.129:30090"
        - name: CLOUD_SERVER_URL
          value: "http://172.16.23.100:8080"
        - name: CHECK_INTERVAL
          value: "5"
```

## Part 2 — Create Service File

Exposes the scheduler on NodePort `30084`. Requests from 5G UEs hit this port — the scheduler handles all routing internally.

```bash
gedit scheduler-service.yaml
```

```yaml
apiVersion: v1
kind: Service
metadata:
  name: scheduler-service
spec:
  type: NodePort
  selector:
    app: scheduler
  ports:
  - port: 5001
    targetPort: 5001
    nodePort: 30084
```

## Part 3 — Deploy

```bash
microk8s kubectl apply -f scheduler-deployment.yaml
microk8s kubectl apply -f scheduler-service.yaml
```

Expected output:
```
deployment.apps/scheduler created
service/scheduler-service created
```

**Watch the pod come up:**

```bash
microk8s kubectl get pods -w | grep scheduler
```

Expected:
```
scheduler-xxxxxxxxx-xxxxx   0/1   ContainerCreating   0   5s
scheduler-xxxxxxxxx-xxxxx   1/1   Running             0   20s
```

**Verify the scheduler started correctly (check startup logs):**

```bash
microk8s kubectl logs -f deployment/scheduler
```

Expected output:
```
===== AI SCHEDULER STARTED =====
Loaded BACKENDS: ['http://172.16.23.129:30081', ...]
Cloud Inference: http://172.16.23.100:8080
================================
Using Device: cpu
Model & Scaler Loaded Successfully
```

> If you see `Running PURE HEURISTIC Mode` instead, the DQN model file was not found in the container. The scheduler still works — it falls back to CPU/MEM/queue-based heuristic routing.

From any machine that can reach the MicroK8s VM:


## Endpoint Reference

| Service | NodePort | Endpoint | Purpose |
|---------|----------|----------|---------|
| Scheduler | `30084` | `POST /predict` | Send image for classification |
| Scheduler | `30084` | `GET /health` | Check scheduler + known nodes |
| Edge Node 1–9 | `30081–30090` | `POST /predict` | Direct edge inference (bypass scheduler) |
| Edge Node 1–9 | `30081–30090` | `GET /metrics` | CPU / MEM / queue stats polled by scheduler |

## Troubleshooting



### `known_nodes: 0` in /health — scheduler can't reach edge nodes

```bash
# Symptom:
curl http://172.16.23.129:30084/health
# "known_nodes": 0

# Diagnosis — test reachability from inside the scheduler pod:
microk8s kubectl exec -it deployment/scheduler -- \
  curl http://172.16.23.129:30081/metrics

# Fix — verify edge pods are running:
microk8s kubectl get pods | grep edge

# Fix — verify BACKEND_URLS env var is set correctly:
microk8s kubectl exec deployment/scheduler -- env | grep BACKEND
```

### All requests going to Cloud (edge nodes never selected)

```bash
# Symptom in logs:
# [OVERLOAD] All Edge Busy → CLOUD

# Diagnosis — check edge node metrics directly:
curl http://172.16.23.129:30081/metrics

# Fix — thresholds are hardcoded at CPU 0.80 / MEM 0.85
# If edges are genuinely idle but metrics show high values,
# check if edge app /metrics endpoint returns correct format
```

### `CLOUD_SERVER_URL` not reachable

```bash
# Diagnosis:
microk8s kubectl exec -it deployment/scheduler -- \
  curl http://172.16.23.100:8080

# Fix — verify Cloud Storage VM is running and port 8080 is open:
# (on Cloud VM) sudo ss -tlnp | grep 8080
```
